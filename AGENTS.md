# AGENTS.md — terraform-provider-anthropic

Guidance for AI agents (and humans) working in this repository.

## What this is

A Terraform provider (`frank-bee/anthropic`) wrapping Anthropic's Managed
Agents, Skills, Environments, Workspaces, and Organization APIs.

## Layout

```
internal/
  apiclient/         # oapi-codegen client + hand-written extensions
    api.yaml         #   OpenAPI spec — edit here first
    apiclient.gen.go #   generated, DO NOT EDIT
    extensions.go    #   hand-written UnmarshalJSON etc. for API quirks
    skills.go        #   hand-written skills client (multipart upload)
    generate.go      #   //go:generate directive
  provider/          # Terraform resources + data sources + tests
  acctest/           # shared test harness
examples/            # source for docs/examples — tfplugindocs consumes this
docs/                # GENERATED by tfplugindocs — do NOT hand-edit
```

## Common commands

| Task | Command |
|------|---------|
| Build binary | `go build ./...` |
| Install into `$GOBIN` (for `~/.terraformrc` dev_overrides) | `go install .` |
| Regenerate API client after editing `api.yaml` | `cd internal/apiclient && go generate ./...` |
| Regenerate docs after touching schema or examples | `go generate ./...` (runs tfplugindocs) |
| Unit tests | `go test ./...` |
| Acceptance tests (real API) | `TF_ACC=1 ANTHROPIC_API_KEY=sk-... go test ./internal/provider/` |
| **Sweep test fixtures after acceptance tests** ⚠️ | `TF_ACC=1 ANTHROPIC_API_KEY=sk-... go test ./internal/provider/ -sweep=all -sweep-allow-failures` |
| `gofmt` the tree (required; CI runs the linter) | `gofmt -s -w -e .` |
| Release | see `RELEASING.md` (tag `vX.Y.Z` → push tag → GoReleaser) |

## MUST-knows when editing

### 1. Always run the sweeper after acceptance tests

Acceptance tests create real resources in the Anthropic API named with a
`tf-` prefix (agents, skills, environments). Normally they're destroyed at
end of each step, but if a DELETE call errors mid-run (wrong beta header,
network blip, etc.) the resource leaks.

**After any `TF_ACC=1 go test` run — success OR failure — run the sweeper:**

```bash
TF_ACC=1 ANTHROPIC_API_KEY=sk-... go test ./internal/provider/ -sweep=all -sweep-allow-failures
```

The sweepers are defined per-resource in `resource_*_test.go`. Each one
deletes every resource whose `Name` / `display_title` starts with `tf-`.

If you don't run this, orphans accumulate in the workspace under names like
`tf-agent-<random>`. They're visible in
`https://platform.claude.com/workspaces/.../agents` and cost inventory space.

### 2. The `anthropic-beta` header choice

The provider sends `anthropic-beta: agent-api-2026-03-01`.

There is a newer `managed-agents-2026-04-01` beta but:
- `DELETE /v1/agents/{id}` on the new beta returns 400 "add
  `agent-api-2026-03-01` to the `anthropic-beta` header".
- The two betas cannot be combined — API explicitly rejects
  `agent-api-*,managed-agents-*` in one header.
- Response shapes differ between the two (string vs object `model`,
  string vs int `version`, different canonical tool type names like
  `agent_toolset_20251212` vs `agent_toolset_20260401`).

`internal/apiclient/extensions.go` adds tolerant `UnmarshalJSON` on
`AgentModelConfig` and `Agent` so the client survives either response
shape even if we switch betas later — don't delete it.

**If you switch betas**, test creates, updates, deletes, and reads
against a real agent first. The provider's acceptance tests will catch
beta-incompatible changes.

### 3. OpenAPI spec is the source of truth

Edit `internal/apiclient/api.yaml` first. Never hand-edit
`apiclient.gen.go` — it's regenerated and your changes will be lost.

After editing `api.yaml`:
```bash
cd internal/apiclient && go generate ./...
cd ../.. && go build ./... && go vet ./...
```

If a field's wire format doesn't match the spec (e.g. string-or-object
polymorphism), add a custom `UnmarshalJSON` in `extensions.go` rather
than hacking the generated file.

### 4. `gofmt` is enforced by CI

The `Release` workflow runs `golangci-lint` which includes a `gofmt` check.
**Run `gofmt -s -w -e .` before every commit** or the release tag will
fail its test job and no binaries will be built.

### 5. Docs are generated

Don't hand-edit `docs/**/*.md`. Edit the schema (`resource_*.go`) and the
examples (`examples/resources/<name>/resource.tf`) — then run
`go generate ./...` at the repo root and commit the regenerated `docs/`.

## Release process (summary)

Full details: `RELEASING.md`. TL;DR:

1. Main is green (Tests workflow passes).
2. `git tag vX.Y.Z && git push origin vX.Y.Z` — pipeline runs tests then
   GoReleaser.
3. Wait ~3-5 min for the release to publish + Terraform Registry to pick it up.
4. Verify: `curl -s https://registry.terraform.io/v1/providers/frank-bee/anthropic/versions | jq '.versions[].version'` should list the new version.

Semver: new attributes = minor bump, fixes = patch bump, removed /
renamed fields = major bump.

## Local testing against a downstream Terraform repo

1. `go install .` to drop the binary in `$GOBIN` (usually `~/go/bin`).
2. Ensure `~/.terraformrc` contains:
   ```hcl
   provider_installation {
     dev_overrides {
       "frank-bee/anthropic" = "/Users/<you>/go/bin"
     }
     direct {}
   }
   ```
3. `terraform plan` in the downstream repo uses the dev-override binary.
   Note the warning Terraform prints — lockfile is not touched in this
   mode.
4. Before committing any lockfile update in the downstream repo, **remove
   / comment out `dev_overrides`**, run `terraform init -upgrade`, then
   restore `dev_overrides`. Otherwise the lockfile won't reflect the
   released provider.

## Known gotchas

- The Anthropic API treats different beta headers almost like different
  API versions — same endpoint, different response shape. Always pin the
  beta header you're testing against.
- `agent_toolset_*` type names are beta-specific:
  `agent-api-2026-03-01` uses `agent_toolset_20251212`, and
  `managed-agents-2026-04-01` uses `agent_toolset_20260401`.
- The skills client uses its own multipart/form-data path (`skills.go`)
  distinct from the oapi-codegen client. It sets its own beta header
  (`skills-2025-10-02`). Don't collapse them.
- Skill source dirs: the `source_dir` attribute uploads an entire tree.
  `SKILL.md` must be at the root; `references/` is the conventional place
  for templates and example files the agent loads at runtime.
